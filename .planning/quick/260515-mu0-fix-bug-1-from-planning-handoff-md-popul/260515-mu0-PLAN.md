<objective>
Fix two production bugs from the post-deploy diagnostic session (.planning/HANDOFF.md) in a single redeploy window. Both bugs surface the same UX symptom — the UI cannot show or act on updates — but live in different packages:

**BUG-1** (internal/docker/discovery.go) — `state.Container.CurrentDigest` is never populated, so the Phase 3 flip-rule in `internal/poll/poller.go` can never flip `UpdateAvailable=true`. Production proof: flutter running local image ID `…18136d85…` while registry holds `…b64c35a5…`, but UI shows no update available. Root cause: `discovery.upsertFromInspect` reads `ContainerInspect.Image` (local content-addressable image ID — NOT the manifest digest the registry returns) and never resolves the registry-shaped digest that lives in `ImageInspect.RepoDigests[0]`.

**BUG-5** (internal/actions/orchestrator.go::drainPullStream) — When the operator clicks Update on a container whose `:latest` tag is already up to date, the daemon returns three Status messages with NO aux payload:
- `{"status":"Pulling from <repo>","id":"latest"}`
- `{"status":"Digest: sha256:<hex>"}`
- `{"status":"Status: Image is up to date for <ref>"}`

`drainPullStream` only scans `aux.ID` / `aux.Digest`, so the function returns `errors.New("docker pull stream ended without aux digest")` and the orchestrator surfaces `action.pull_failed`. Production proof: 16:26:34 logs show this error on every Update / Force-pull for seatd and centroidx-backend; manual `docker pull` on the same daemon (29.4.3 / API 1.54) works fine — same stream shape. The bug is purely in our parser.

Purpose: unblock BUG-4 (`update_available` flip — automatic green once BUG-1 lands) AND restore Update-button functionality for already-up-to-date images (BUG-5). Both fixes ship in one redeploy so the next manual smoke on the HMI box validates BOTH paths at once.
