## D-N1V-01 — Watched-container labels stay namespaced `hmi-update.*`

**Decision:** The label keys `hmi-update.watch`, `hmi-update.allow-update`, `hmi-update.allow-rollback`, `hmi-update.tag-pattern`, `hmi-update.wait-for-healthy` are **NOT** renamed by this plan.

**Cost of renaming (rejected option):** Operators across the Centroid HMI fleet already have these labels on dozens of compose service blocks (`flutter`, `centroidx-backend`, `weston`, `seatd`, `timescaledb`, cert/init containers, and any others). A label-namespace rename would force a synchronized edit on every HMI's `docker-compose.yml`, coordinated with the binary upgrade. Miss a single label on a single HMI and that container silently stops being watched — the binary cannot warn, because "labels absent" is the legitimate signal for "not a watched container."

**Cost of preserving (chosen option):** A permanent visible inconsistency between the operator-facing name (`docker-update`) and the label-namespace prefix (`hmi-update.`). Documented in README.md, CLAUDE.md, and PROJECT.md as a deliberate backwards-compat contract; future contributors get the rationale inline at the load-bearing code sites (`internal/docker/discovery.go` filter, `internal/actions/middleware.go` constants, `ui/src/lib/Row.svelte` derives).

**Outcome:** Locked. Re-evaluate only if a future milestone introduces a one-shot migration tool that can edit all HMI compose files atomically.

## D-N1V-02 — Env-var prefix renames in lockstep with the binary

**Decision:** Every `HMI_UPDATE_*` env var renames to `DOCKER_UPDATE_*` in a single task commit (Task 3). No transitional dual-read.

**Why not transitional dual-read:** Adds permanent code surface (every `os.Getenv` site becomes a two-line fallback function). The HMI fleet is small (currently one box at 10.50.10.175 plus the imminent fleet additions) — coordinated upgrade is cheaper than permanent ambiguity.

**Migration note for operators:** README.md adds an "Upgrading from hmi-update" section listing the env-var renames + state-file path rename + service-name rename, with a single-paragraph compose-snippet that shows the before/after side by side.

## D-N1V-03 — State file renames `hmi_update_state.json` -> `docker_update_state.json`

**Decision:** Default state-file path renames. The on-disk schema is unchanged — the file is JSON with the same fields; only the filename + bind-mount path change.

**Migration for the existing HMI at 10.50.10.175:** Coordinated separately with the redeploy:
```

```
Documented in README.md "Upgrading from hmi-update."
