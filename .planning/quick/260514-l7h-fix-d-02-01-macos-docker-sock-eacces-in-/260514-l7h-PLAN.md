<objective>
Fix D-02-01: hmi-update's e2e base stack fails on macOS Docker Desktop because the in-container UID 65532:65532 has no permission on the host-forwarded `/var/run/docker.sock` (root:root, mode 0660 inside the LinuxKit VM). On Linux the docker group GID varies per distro and is also not 65532. The current base `compose.test.yml` has no `user:` override, so the container inherits the Dockerfile's nonroot user (65532:65532), which is never in the host docker group.

This blocker prevents `make e2e` from passing on Apple Silicon dev machines and any Linux host whose docker GID != 65532. Phase 02 left a TODO line in STATE.md ("Fix belongs in Phase 1 or Phase 7") — this quick plan closes it now so Phase 03 development can proceed on the developer's macOS host.

Purpose: Make the positive (base) e2e path grant the container the host's docker GID at compose-up time, while preserving the negative (eacces override) path's deterministic 503 behaviour.

Output:
  - `e2e/compose.test.yml` carries `user: "65532:${HMI_DOCKER_GID:-65532}"` on the hmi-update service.
  - `Makefile` `e2e` + `e2e-debug` targets export `HMI_DOCKER_GID` from the host's docker.sock GID before `docker compose up`, portably across GNU and BSD `stat`.
  - `e2e/compose.test.override.eacces.yml` updated comments to clarify the new HMI_DOCKER_GID base var is overridden literally by `65532:65532` (scalar-merge wins).
</objective>

Compose v2 scalar-merge: when both base and override declare `user:`, the override's literal string wins. So once the base gains `user: "65532:${HMI_DOCKER_GID:-65532}"`, the override's `"65532:65532"` REPLACES it entirely — env var is not consulted. EACCES path remains deterministic.

The macOS twist (not in PITFALLS.md, captured in CONTEXT below from STATE.md D-02-01 + the planning context):
On macOS Docker Desktop the host has no `docker` group at all (`id -g docker` returns nothing meaningful); the socket is forwarded via the LinuxKit VM where it's owned by root:root mode 0660. The fix is to read the GID directly off the bind-mount target via `stat`. `stat -c '%g'` (GNU/Linux) vs `stat -f '%g'` (BSD/macOS) — chain with `||`.

**1. `e2e/compose.test.yml` — add `user:` to the hmi-update service.**

Insert a `user:` field on the `hmi-update` service block (anywhere between `build:` and `depends_on:` is fine — recommend right after `ports:` for readability). Use the exact form:

```yaml
    # macOS Docker Desktop + Linux dual support. The container's effective
    # UID stays at 65532 (distroless nonroot reserved). The GID is overridden
    # at compose-up time from HMI_DOCKER_GID, which the Makefile derives from
    # `stat`-ing /var/run/docker.sock on the host (see Makefile `e2e` target).
    # Fallback to 65532 keeps explicit failure visible when the env var is
    # unset: /healthz will return 503 EACCES with the Pitfall 9 hint, the
    # same way the compose.test.override.eacces.yml override does
    # deliberately. Override files that pin a literal user: (e.g.
    # compose.test.override.eacces.yml's "65532:65532") bypass this base
    # entry — compose v2 scalar-merge replaces the whole string.
    user: "65532:${HMI_DOCKER_GID:-65532}"
```

Keep all other fields (volumes, tmpfs, environment, depends_on, ports) intact and in their current order. Do NOT touch any other service block.

**2. `Makefile` — detect docker.sock GID and export it before `docker compose up`.**

Update BOTH the `e2e` and `e2e-debug` targets. The detection has to be cross-platform: GNU `stat -c '%g'` and BSD `stat -f '%g'` are mutually exclusive — try GNU first, fall back to BSD; if both fail (no docker.sock at all) leave the var unset and let the compose default of 65532 surface as a deterministic EACCES failure with the documented remediation hint.

Use a make-variable + recipe-line export pattern (works in GNU Make, the only Make assumed by this project's CI). The detection must run at recipe-execution time (not parse time) so a developer who starts Docker Desktop AFTER cloning the repo still gets the right GID.

Replace the existing `e2e:` target body with:

```make
e2e:
	cd e2e && npm ci && npx playwright install --with-deps chromium
	HMI_DOCKER_GID=$$(stat -c '%g' /var/run/docker.sock 2>/dev/null || stat -f '%g' /var/run/docker.sock 2>/dev/null) ; \
	  echo "[make e2e] HMI_DOCKER_GID=$${HMI_DOCKER_GID:-<unset; container will hit EACCES>}" ; \
	  HMI_DOCKER_GID=$$HMI_DOCKER_GID docker compose -f e2e/compose.test.yml up -d --wait
	cd e2e && npx playwright test ; STATUS=$$? ; \
	  docker compose -f e2e/compose.test.yml down -v --remove-orphans ; \
	  exit $$STATUS
```

Replace the existing `e2e-debug:` target body with the same `HMI_DOCKER_GID` detection prefix on the `up` line:

```make
e2e-debug:
	cd e2e && npm ci && npx playwright install --with-deps chromium
	HMI_DOCKER_GID=$$(stat -c '%g' /var/run/docker.sock 2>/dev/null || stat -f '%g' /var/run/docker.sock 2>/dev/null) ; \
	  echo "[make e2e-debug] HMI_DOCKER_GID=$${HMI_DOCKER_GID:-<unset; container will hit EACCES>}" ; \
	  HMI_DOCKER_GID=$$HMI_DOCKER_GID docker compose -f e2e/compose.test.yml -f e2e/compose.test.override.debug.yml up -d --wait --build
	cd e2e && npx playwright test ; STATUS=$$? ; \
	  docker compose -f e2e/compose.test.yml down -v --remove-orphans ; \
	  exit $$STATUS
```

Important shell mechanics:
- `$$` in a Make recipe escapes to a single `$` for the shell — required for `$(stat …)` and `$HMI_DOCKER_GID`.
- The trailing `;` + `\` continuations keep the whole sequence in ONE shell invocation so the var survives from the detection line to the compose-up line. Without the `;\` they'd run in separate sub-shells and the var would be lost.
- The `2>/dev/null` swallows the OS-specific "invalid option" stderr (GNU stat barfs on `-f`, BSD stat barfs on `-c`).
- The `echo` line is a deliberate diagnostic — when this breaks for some user on some new platform, they'll see exactly what value got injected.

DO NOT alter the `build:`, `test:`, `ui:`, `types:`, `check-types:`, `image:`, `image-debug:`, or `clean:` targets. They have no docker.sock interaction.
