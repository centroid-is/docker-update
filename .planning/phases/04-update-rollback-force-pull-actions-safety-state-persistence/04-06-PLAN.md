    ```typescript
    // e2e/fixtures/disconnect-network.ts
    //
    // ACT-04: Rollback MUST work with the registry network detached. This
    // fixture wraps `docker network disconnect` so the rollback-flow spec
    // can simulate an offline HMI.
    //
    // Pattern matches push-image.ts (execSync child_process).

    import { execSync } from 'node:child_process';

    function getComposeNetwork(): string {
      const networks = execSync(`docker network ls --format '{{.Name}}'`, { encoding: 'utf8' });
      const match = networks.split('\n').find((n) => /e2e.*_default$/.test(n));
      if (!match) {
        throw new Error(`Could not find e2e compose network in:\n${networks}\nIs the stack up?`);
      }
      return match;
    }

    export function disconnectZotFromNetwork(): void {
      const net = getComposeNetwork();
      execSync(`docker network disconnect ${net} zot`, { stdio: 'inherit' });
    }

    export function reconnectZot(): void {
      const net = getComposeNetwork();
      execSync(`docker network connect ${net} zot`, { stdio: 'inherit' });
    }
    ```

Acceptable. Document this in the file header: "Network name derives from `docker network ls` output filtered by /e2e.*_default$/; not from operator input. execSync is safe here. Future evolution: pivot to execFileSync if network names ever become operator-supplied."

    Note on watch=false: the verify-failed.spec.ts test needs the container to exist (so the orchestrator can POST against its service name), but NOT to be in the Discoverer's watch set (because the crash-loop would generate spurious events in unrelated specs). The test will POST against an explicit service name "crash-loop-stub"; if the orchestrator's LookupContainer middleware (Plan 04-03) returns 404 for "crash-loop-stub" because it's not in state, the test fails the WRONG way.

    Resolution: the orchestrator's LookupContainer reads from cached state populated by the Discoverer; if crash-loop-stub is not in the watch set it won't appear in state and the test would get 404 (not the expected 500 verify_failed). So crash-loop-stub MUST be watched. The expected noise in other tests: the crash-loop generates docker events but those events for a container without `hmi-update.watch=true` are filtered by the Discoverer's label filter. With watch=true the events ARE captured.

    Final decision: set `hmi-update.watch: "true"` and document the intentional crash-loop in the verify-failed.spec.ts file header so future developers understand why a "broken" container is part of the compose stack:

    ```yaml
      command: ["sh", "-c", "exit 1"]
      restart: unless-stopped
      labels:
        hmi-update.watch: "true"
    ```

    The Discoverer will see this container's `die` events and mark it `Stopped: true` in state, which is fine — the verify-failed.spec.ts test specifically expects the post-Update verify loop to fail because the container keeps crashing.
