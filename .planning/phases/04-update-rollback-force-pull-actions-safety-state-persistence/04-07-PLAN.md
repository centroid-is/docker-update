## Recommended Approach: Option B (orchestrator refactor — crane.Pull → docker.ImageLoad)

**The architecturally cleanest fix.** The orchestrator currently leans on `docker.Client.ImagePull(ref)`, which mandates the daemon perform name resolution. By having the orchestrator drive image acquisition itself — using `google/go-containerregistry`'s `crane.Pull` (which already powers Phase 3's `registry.Resolver`) to fetch the manifest + layers via the in-container HTTP path, then `docker.Client.ImageLoad` to write the resulting image tarball into the daemon's local cache — we decouple image acquisition from the daemon's DNS context entirely.

### Why this beats the alternatives

| Option | What it changes | Pros | Cons |
|--------|----------------|------|------|
| **A (host-side extra_hosts hack)** | Adds `extra_hosts: ["zot:host-gateway"]` to the e2e compose stack and pivots image refs to `localhost:15000`. | Smallest code surface change. Zero production-code modification. | Pushes the test-harness divergence further — production talks to `ghcr.io` via the container network, test talks to `localhost` via a host port-forward. Two networking models under one harness; a future bug that exists in one path but not the other is harder to catch. |
| **B (crane.Pull + ImageLoad refactor — RECOMMENDED)** | Replaces `docker.Client.ImagePull` inside the orchestrator with `crane.Pull` (already imported via `google/go-containerregistry`) followed by `docker.Client.ImageLoad`. | Production path now uses the SAME HTTP client the resolver already authenticated and redacted. Removes the daemon's role in name resolution entirely. Unifies the bearer-token + redactingTransport story (Phase 3 OBS-04 carry-forward — no need to also redact daemon-level pull logs). Cuts one moving part out of the offline-rollback story (ImageTag already operates against the local cache; ImageLoad is the same shape). | ~80-120 LOC orchestrator change. Need to handle tarball assembly (`crane.Pull` returns `v1.Image`; we use `crane.Save` or `tarball.Write` to feed `docker.Client.ImageLoad`). Behaviour-equivalent to ImagePull but the wire path is different — must add unit tests pinning the new path. |
| **C (Docker Desktop daemon.json registry-mirrors)** | Configures the dev/CI docker daemon to mirror `zot:5000` to `localhost:15000`. | Zero code change. | Not portable across CI workers; requires per-machine daemon config; doesn't generalize to Linux CI runners that don't use Docker Desktop. |

### Option B implementation sketch

```go
// internal/actions/orchestrator.go (new helper, replacing direct ImagePull)
func (o *actionOrchestrator) acquireImage(ctx context.Context, ref string, digest string) error {
    // Step 1: fast path — does the daemon already have this digest? ImageInspect.
    if _, _, err := o.docker.ImageInspectWithRaw(ctx, ref+"@"+digest); err == nil {
        return nil // local cache hit; nothing to do
    }
    // Step 2: pull via crane (in-container HTTP — uses Phase 3 redactingTransport).
    img, err := crane.Pull(ref, o.craneOpts()...)
    if err != nil {
        return fmt.Errorf("crane.Pull %s: %w", ref, err)
    }
    // Step 3: stream the image tarball into the daemon's cache via ImageLoad.
    rdr, wtr := io.Pipe()
    go func() {
        defer wtr.Close()
        _ = tarball.Write(wtr, ref, img) // single-tag tarball
    }()
    resp, err := o.docker.ImageLoad(ctx, rdr, true /* quiet */)
    if err != nil {
        return fmt.Errorf("docker.ImageLoad: %w", err)
    }
    defer resp.Body.Close()
    _, _ = io.Copy(io.Discard, resp.Body) // drain progress JSON
    return nil
}
```

**Key invariants preserved:**
- The crane.Pull call uses the same `authn.Keychain` + `redactingTransport` Phase 3 wired into `registry.Resolver` — no double-auth, no token leak surface change.
- Local cache hit (step 1) makes offline rollback (ACT-04) trivially correct: the digest is already present from the prior Update, so we skip straight past pull/load.
- `compose.Runner.UpdateService` is unchanged — it still drives `docker compose up -d --force-recreate <svc>`, which finds the image in the local cache regardless of how it got there.
