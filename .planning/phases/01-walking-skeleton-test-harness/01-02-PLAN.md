    Implement per RESEARCH.md section "renameio/v2 API and the directory-fsync correction" verbatim wrapper (lines 478-513). The code is reproduced here for executor convenience - copy the comment header verbatim so the rationale survives future cleanup PRs:

    ```go
    package state

    import (
        "encoding/json"
        "os"
        "path/filepath"

        "github.com/google/renameio/v2"
    )

    // persist writes the in-memory state to disk atomically.
    // renameio.WriteFile handles temp-file-in-same-dir + fsync(file) + rename.
    // It does NOT fsync the parent directory; we do that explicitly to make the
    // rename durable across host crash, not just process crash.
    // See renameio issue #11 and research/PITFALLS.md Pitfall 7
    // (also 01-RESEARCH.md research correction A5).
    //
    // Caller invariant: must be called with s.mu held in write mode.
    func (s *Store) persist() error {
        data, err := json.MarshalIndent(s.state, "", "  ")
        if err != nil {
            return err
        }
        if err := renameio.WriteFile(s.path, data, 0o644); err != nil {
            return err
        }
        // Best-effort directory fsync. If this fails the rename is still
        // visible to subsequent opens; we only lose durability across power loss.
        dir, err := os.Open(filepath.Dir(s.path))
        if err != nil {
            return nil
        }
        defer dir.Close()
        _ = dir.Sync()
        return nil
    }
    ```

    Decision logged per A5: shipping Option 2 (the wrapper) - chosen because (a) ~50us per write is invisible at this scale, (b) the brief implies durability across operator-action crashes on HMIs, (c) Phase 4's SIGKILL fault test would have flaked without it.
