<objective>
Ship `.github/workflows/publish.yml` — the publish workflow that fires after `ci.yml` passes
on `main` (via `workflow_run`) or on a Git tag push matching `v*.*.*`. The workflow builds the
production image via Phase 7's Dockerfile, tags it with the three canonical tags
(`:latest`, `:vX.Y.Z`, `:sha-<short>`) via `docker/metadata-action@v5`, pushes to
`ghcr.io/centroid-is/hmi-update` via `docker/build-push-action@v6`, and then runs the
post-publish real-GHCR anonymous-token-flow smoke against BOTH a frozen anchor AND the
just-pushed image.

Purpose: This plan delivers Phase 8's headline output. After this plan ships, a green commit
on `main` automatically produces an image at `ghcr.io/centroid-is/hmi-update` with all three
tags; an operator can `docker pull ghcr.io/centroid-is/hmi-update:sha-<short>` for the manual
smoke step documented in 08-03's `RELEASING.md`.

The post-publish smoke is the BELT-AND-BRACES counterpart to ci.yml's `ghcr-smoke-pr`. The PR
smoke catches the regression before merge; the post-publish smoke catches the case where
GHCR's visibility/auth flow for the freshly-pushed image regresses (e.g. an org-level
visibility flip, a registry-side change that breaks the anonymous read of OUR image
specifically rather than the frozen anchor).

Output: One new workflow file; a recorded test run on a feature branch proving the publish
fires on `main` and emits the expected three tags; a recorded test run on a `v0.0.0-test`
tag (pre-release semver) proving the tag-push path also fires.
</objective>

Triggers:
- `workflow_run` of workflow `"ci"`, `types: [completed]`, `branches: [main]`. The `if:` guard
  on the publish job filters to `github.event.workflow_run.conclusion == 'success'`.
- `push: tags: ['v*.*.*']`. Fires on a Git semver tag push regardless of branch ancestry; the
  tag is the source of truth for "this is a release."

Job graph:
- `publish`: depends on the trigger; uses metadata-action + login-action + build-push-action.
  Outputs: `primary_tag` (the first tag emitted by metadata-action — `latest` for main pushes,
  `<version>` for tag pushes).
- `ghcr-smoke-published`: needs [publish]; declares `if: success()`; anonymous-token-flow
  assertions identical to 08-01's `ghcr-smoke-pr` BUT also probes the just-pushed
  `ghcr.io/centroid-is/hmi-update:${{ needs.publish.outputs.primary_tag }}`.

The `ghcr-smoke-pr` job in 08-01's `ci.yml` is the PR-side defense; this plan's
`ghcr-smoke-published` is the post-publish defense. The TWO smokes together cover:
- 08-01: pre-merge regression in `internal/registry` or `go-containerregistry` dependency bump.
- 08-02: post-merge regression in GHCR's anonymous read of OUR org repo specifically.

Output emitted by the publish job for downstream consumption:
```yaml
outputs:
  primary_tag: ${{ steps.parse-meta.outputs.primary_tag }}
  digest: ${{ steps.build.outputs.digest }}
```

`primary_tag` is the first line of `steps.meta.outputs.tags` (which is a newline-separated
list of fully-qualified `ghcr.io/centroid-is/hmi-update:<tag>` strings). The `parse-meta`
step strips the `ghcr.io/centroid-is/hmi-update:` prefix to leave just the tag string.

  <action>
    Create `.github/workflows/publish.yml`:

    ```yaml
    name: publish

    on:
      workflow_run:
        workflows: ["ci"]
        types: [completed]
        branches: [main]
      push:
        tags: ['v*.*.*']

    permissions:
      contents: read

    concurrency:
      group: publish-${{ github.ref }}
      cancel-in-progress: true

    jobs:
      publish:
        # Triggered either by a successful ci.yml on main OR by a semver tag push.
        if: |
          (github.event_name == 'workflow_run' && github.event.workflow_run.conclusion == 'success') ||
          (github.event_name == 'push' && startsWith(github.ref, 'refs/tags/v'))
        runs-on: ubuntu-24.04
        permissions:
          contents: read
          packages: write
        outputs:
          primary_tag: ${{ steps.parse-meta.outputs.primary_tag }}
          digest: ${{ steps.build.outputs.digest }}
        steps:
          - uses: actions/checkout@v4
            with:
              # When triggered by workflow_run, use the exact SHA that CI tested,
              # not the (possibly-moved-since) main HEAD. For tag pushes, the
              # default ref (the tag's commit) is what we want.
              ref: ${{ github.event.workflow_run.head_sha || github.ref }}

          - uses: docker/setup-buildx-action@v3

          - uses: docker/login-action@v3
            with:
              registry: ghcr.io
              username: ${{ github.actor }}
              password: ${{ secrets.GITHUB_TOKEN }}

          - id: meta
            uses: docker/metadata-action@v5
            with:
              images: ghcr.io/centroid-is/hmi-update
              tags: |
                type=raw,value=latest,enable={{is_default_branch}}
                type=semver,pattern={{version}}
                type=sha,prefix=sha-,format=short
              labels: |
                org.opencontainers.image.title=hmi-update
                org.opencontainers.image.description=Centroid HMI per-container update and rollback
                org.opencontainers.image.source=https://github.com/centroid-is/hmi-update

          - id: parse-meta
            name: derive primary_tag (first tag emitted by metadata-action)
            run: |
              # metadata-action's tags output is newline-separated, fully-qualified.
              # The first line is the canonical tag to smoke-test against.
              FIRST_FQ=$(echo "${{ steps.meta.outputs.tags }}" | head -n 1)
              # Strip the image-ref prefix so we're left with just the tag string.
              PRIMARY="${FIRST_FQ##ghcr.io/centroid-is/hmi-update:}"
              echo "primary_tag=$PRIMARY" >> "$GITHUB_OUTPUT"
              echo "[parse-meta] primary_tag=$PRIMARY (from $FIRST_FQ)"

          - id: build
            uses: docker/build-push-action@v6
            with:
              context: .
              file: Dockerfile
              platforms: linux/amd64
              push: true
              tags: ${{ steps.meta.outputs.tags }}
              labels: ${{ steps.meta.outputs.labels }}
              cache-from: type=gha
              cache-to: type=gha,mode=max
              provenance: false
              sbom: false

          - name: emit step summary
            run: |
              {
                echo "## publish — hmi-update"
                echo ""
                echo "**Tags pushed:**"
                echo '```'
                echo "${{ steps.meta.outputs.tags }}"
                echo '```'
                echo ""
                echo "**Image digest:** \`${{ steps.build.outputs.digest }}\`"
                echo ""
                echo "**Primary tag (smoke target):** \`${{ steps.parse-meta.outputs.primary_tag }}\`"
              } >> "$GITHUB_STEP_SUMMARY"

      ghcr-smoke-published:
        # Post-publish anonymous-token-flow smoke. Runs ONLY after publish succeeds.
        # See 08-CONTEXT.md Area 3 + 08-RESEARCH.md Question 2.
        needs: [publish]
        if: success()
        runs-on: ubuntu-24.04
        permissions:
          contents: read
        steps:
          - name: assert no credentials in env
            run: |
              if [ -n "$GITHUB_TOKEN" ]; then
                echo "FAIL: ghcr-smoke-published saw GITHUB_TOKEN; bug in job permissions" >&2
                exit 1
              fi
              if [ -f "$HOME/.docker/config.json" ]; then
                echo "FAIL: ghcr-smoke-published saw ~/.docker/config.json; a prior step logged in" >&2
                exit 1
              fi
              echo "OK: no credentials in scope; anonymous bearer-token flow will be exercised"

          - uses: actions/setup-go@v5
            with:
              go-version: '1.26'

          - name: install crane (pinned)
            run: go install github.com/google/go-containerregistry/cmd/crane@v0.20.8

          - name: anonymous digest probe — frozen public anchor
            run: |
              set -e
              DIGEST=$(crane digest ghcr.io/distroless/static-debian12:nonroot)
              echo "frozen-anchor=$DIGEST"
              echo "frozen-anchor=$DIGEST" >> "$GITHUB_STEP_SUMMARY"
              [ -n "$DIGEST" ] || (echo "FAIL: crane digest returned empty string for frozen anchor" >&2; exit 1)

          - name: anonymous digest probe — just-pushed image
            run: |
              set -e
              REF="ghcr.io/centroid-is/hmi-update:${{ needs.publish.outputs.primary_tag }}"
              echo "[ghcr-smoke-published] probing $REF anonymously"
              DIGEST=$(crane digest "$REF")
              echo "just-pushed=$DIGEST"
              echo "just-pushed=$REF@$DIGEST" >> "$GITHUB_STEP_SUMMARY"
              [ -n "$DIGEST" ] || (echo "FAIL: crane digest returned empty string for just-pushed image" >&2; exit 1)
              # Cross-check: the digest matches what the publish job recorded.
              EXPECTED="${{ needs.publish.outputs.digest }}"
              if [ -n "$EXPECTED" ] && [ "$DIGEST" != "$EXPECTED" ]; then
                echo "WARN: anonymous-read digest ($DIGEST) differs from build-push digest ($EXPECTED)" >&2
                echo "  This can happen with multi-arch indexes; investigate before next release." >&2
                # Soft-fail in v1 — log loud, don't fail the workflow. The digest mismatch is
                # informational unless we promise byte-identical pushes (we don't, per
                # 08-CONTEXT.md "Pipeline-build vs. published-image identity").
              fi
    ```

    Constraints to honor (DO NOT deviate without surfacing in <decisions>):

    - **Top-level permissions: `contents: read`.** Job-level escalation only on `publish` (adds `packages: write`).
    - **`ghcr-smoke-published` MUST declare `permissions: contents: read` explicitly.** Belt-and-braces: even though workflow-level is `contents: read`, the explicit job-level declaration documents the "no packages" invariant and prevents a future workflow-level escalation from accidentally giving the smoke job creds.
    - **`workflow_run.head_sha` checkout.** Race-safe: the publish builds from the exact SHA that ci.yml tested green, not a possibly-moved main HEAD. (See CONTEXT.md "Pipeline-build vs. published-image identity.")
    - **`metadata-action@v5` config is byte-identical to 08-RESEARCH.md Question 1.** The three-tag grammar is the locked contract.
    - **`build-push-action@v6` `provenance: false` + `sbom: false`** per CONTEXT.md `<specifics>`. Image-size budget already strained; supply-chain artifacts deferred to V2.
    - **No `workflow_dispatch:` trigger.** CONTEXT.md `<specifics>` decision — no ad-hoc publish button in v1.
    - **`if:` on the publish job covers BOTH triggers.** `workflow_run` path needs `conclusion: success`; tag push path needs `startsWith(github.ref, 'refs/tags/v')`. The OR-combined guard handles both.
    - **`needs.publish.outputs.primary_tag`** is the contract between publish and smoke. The `parse-meta` step's responsibility is to extract a single tag string (no fully-qualified prefix) from metadata-action's tags output.
    - **Concurrency group is `publish-${{ github.ref }}`**, INDEPENDENT from ci.yml's `${{ github.workflow }}-${{ github.ref }}`. Reason: ci.yml and publish.yml have different ref semantics (publish runs against the workflow_run's head_sha, not the live ref); a single group across the two would cancel publishes inappropriately.
    - **The digest cross-check is a SOFT warning, not a fail.** Reason: build-push-action's `outputs.digest` is the manifest digest as built; `crane digest` returns the registry's reported digest. For a single-arch linux/amd64 build they should match, but a future multi-arch flip would make them differ legitimately (one is the index digest, the other is the platform manifest digest). v1 logs the mismatch; v2 may tighten.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `.github/workflows/publish.yml` → GHCR push endpoint | Authenticated with `${{ secrets.GITHUB_TOKEN }}` + `${{ github.actor }}`. Token is short-lived and scoped to the workflow run. |
