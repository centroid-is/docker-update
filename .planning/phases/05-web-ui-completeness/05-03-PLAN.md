<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User input (modal buttons) → onConfirm/onCancel | Modal only fires callbacks; the actual POST happens in Plan 05-04. A keyboard or pointer race producing two Continue clicks is bounded by Plan 05-04's `isActing` flag and the server's per-service mutex (Phase 4 ACT-08). |
| Toast aria-live | Error toasts use `role="alert"`; success/info/warning use `role="status"`. This is the standard SR-announcement boundary. |
| display-warning.ts → service name | A maliciously-named service `not-flutter-but-fooled-you` containing `flutter` substring will trigger the warning — that's the intended behavior (false positive is operator-protective; false negative would be the bad failure mode). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-03-01 | T (Tampering) | DOM modal removal | accept | A user who removes the modal via devtools bypasses the warning but still hits the unguarded POST — the warning is operator-protective UX, not a security boundary. Server runs the recreate regardless. Documented in 05-CONTEXT.md Area 4. |
| T-05-03-02 | I (Info Disclosure) | Toast body content | mitigate | Error toast bodies surface `body.reason` from the server, which Phase 4 already redacts via Phase 3's slog ReplaceAttr / redacting transport. No new exposure. |
| T-05-03-03 | A (Accessibility) | Modal focus trap correctness | mitigate | focus-trap.ts handles Tab + Shift-Tab + Escape. Manual a11y smoke (Plan 05-05 e2e) verifies. |
| T-05-03-04 | D (DoS) | Toast queue growth | mitigate | Toasts auto-dismiss after 5 s (except error, click-to-dismiss). Worst case is N error toasts stacked vertically with no limit — acceptable; an operator who triggers many errors is already in a degraded state and the toasts are the diagnosis. |
| T-05-03-05 | I | Modal body scroll lock leakage | mitigate | Effect cleanup restores `document.body.style.overflow` to its prior value; never to empty (which would have been a bug if there were other modals). |
| T1 (phase-wide) | E | n/a in 05-03 | n/a | T1 (in-place upgrade) mitigated by Plan 05-05. |
</threat_model>
